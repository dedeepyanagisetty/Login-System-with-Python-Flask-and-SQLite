Bro, this is a very important Flask concept 👍

Why do we set FLASK_APP?

When you run:

flask run

Flask asks:

Which file contains my application?

It doesn't know automatically.

So we tell Flask:

$env:FLASK_APP="run.py"

Meaning:

Hey Flask 👋

My application starts from:

run.py

Then Flask internally does something like:

from run import app

or for Factory Pattern:

from run import app

and starts the server.

Without FLASK_APP

If you do:

flask run

Flask may say:

Error: Could not locate a Flask application

because it doesn't know where your app is.

Why does python run.py work without FLASK_APP?

Because:

python run.py

directly executes:

app = create_app()

if __name__ == "__main__":
    app.run()

Python already knows which file to run.

No discovery needed.

Why use flask run then?

Because Flask gives extra development features:

flask run --debug

Benefits:

Auto Reload
Debugger
Environment Variable Support
Flask Commands
Flask-Migrate Commands
Dynamic Host

Example:

flask run --host=0.0.0.0

Meaning:

Listen on all network interfaces

So:

127.0.0.1
192.168.x.x
localhost

all can access.

Dynamic Port

Example:

flask run --port=5001

Meaning:

Use port 5001
instead of
5000

URLs become:

http://127.0.0.1:5001

instead of:

http://127.0.0.1:5000
Combined
flask run --host=0.0.0.0 --port=5001 --debug

Means:

Host      = 0.0.0.0
Port      = 5001
Debug     = True
AutoReload= True
Production (Render)

You normally DON'T do:

flask run

Instead:

gunicorn run:app

because:

flask run  -> Development Server
gunicorn   -> Production Server
